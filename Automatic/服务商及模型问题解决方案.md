# 问题与解决方案总结

## 一、服务商修改后只有硅基流动可用，其余不能调用

### 问题描述
修改了项目中的默认服务商配置后，只有硅基流动可用，其余服务商都不能调用。

### 原因分析
1. **API 密钥配置问题**：所有服务商的 API 密钥都没有设置（都是空字符串）
2. **配置变更**：修改了 `providers.ts` 文件中的服务商配置，但 `.env` 文件中的 API 地址仍然指向原来的地址
3. **代码逻辑**：对于 Embedding 模型，即使没有 API 密钥，它也会使用 fallback 模型列表，所以硅基流动的 Embedding 模型可以正常工作；对于其他服务商的文本模型，`getLLM` 函数会尝试获取 API 密钥，如果没有获取到，会抛出错误

### 解决方案
1. **更新 .env 文件**：将所有服务商的 API 地址和模型名称更新为新的配置
2. **设置 API 密钥**：在 `.env` 文件中为需要使用的服务商设置 API 密钥
3. **修改默认配置**：如果暂时没有 API 密钥，可以修改 `rag.ts` 文件中的 `asEmbeddingProvider` 函数，让它默认返回 "openai"，或者修改 `providerPriority` 数组的顺序，让 openai 优先使用

## 二、填写 API 密钥测试提示认证失败

### 问题描述
填写腾讯混元的 API 密钥并测试时，系统提示 "401 Authentication Fails, Your api key: ***9kFG is invalid"。

### 原因分析
当系统尝试使用腾讯混元的 API 密钥去访问 DeepSeek 的 API 地址（https://api.deepseek.com/v1），而不是腾讯混元的 API 地址（https://api.hunyuan.cloud.tencent.com/v1）。这是因为虽然修改了 `providers.ts` 文件中的配置，但 `.env` 文件中的 `DEEPSEEK_BASE_URL` 仍然指向原来的 DeepSeek API 地址。

### 解决方案
更新 `.env` 文件中的所有服务商配置，确保它们与 `providers.ts` 文件中的配置一致：

1. **腾讯混元**：
   - API 地址：`https://api.hunyuan.cloud.tencent.com/v1`
   - 模型：`hunyuan-lite`

2. **其他服务商**：
   - 也都更新了相应的 API 地址和模型名称

## 三、保存配置时提示"请求参数校验失败"

### 问题描述
在模型路由管理台保存配置时，系统提示"请求参数校验失败"。

### 原因分析
1. 前端的 `RouteDraft` 接口中 `temperature` 和 `maxTokens` 字段是字符串类型
2. 在保存时，系统使用 `Number()` 函数将这些字符串转换为数字
3. 如果用户输入了非数字字符（或者输入框为空），`Number()` 会返回 `NaN`
4. 后端的验证规则要求这些字段必须是有效的数字，因此会校验失败
5. 此外，当 `maxTokens` 为 `null` 时，后端的验证规则也会拒绝该参数

### 解决方案
1. **修改前端代码**，确保在保存模型路由时：
   - 使用 `parseFloat()` 函数来转换 `temperature` 字段
   - 使用 `parseInt()` 函数来转换 `maxTokens` 字段
   - 使用 `isNaN()` 函数检查转换结果是否为有效数字
   - 如果转换失败，使用默认值（`temperature` 为 0.7，`maxTokens` 为 null）

2. **修改后端验证规则**，允许 `maxTokens` 为 `null`：
   - 在 `modelRouteUpsertSchema` 中，将 `maxTokens` 的验证规则从 `z.number().int().min(64).max(16384).optional()` 修改为 `z.union([z.number().int().min(64).max(16384), z.null()]).optional()`

### 修改的文件
- `client/src/pages/settings/ModelRoutesPage.tsx` - 修复了保存按钮的点击事件处理逻辑
- `server/src/routes/llm.ts` - 修改了模型路由的验证规则，允许 `maxTokens` 为 `null`

## 四、创建小说项目时使用AI生成内容时提示Schema校验失败

### 问题描述
创建小说项目并使用 AI 生成内容时，系统提示 "未通过 Schema 校验，错误：(root): Invalid input: expected object, received array"。

### 原因分析
1. `noveldirector.candidates@v1` Schema 期望的是一个包含 `candidates` 字段的对象，该字段是一个数组
2. 但 AI 生成的内容直接是一个数组，没有包装在 `candidates` 字段中
3. 当 JSON 修复后，仍然是一个数组，导致 Schema 校验失败

### 解决方案
修改 `repairWithLlm` 函数，添加特殊处理逻辑：

1. 如果解析到的是数组，尝试将其包装成 `{ candidates: array }` 的形式
2. 然后检查这种包装形式是否能通过 Schema 校验
3. 如果能通过，就使用这种包装形式

### 修改的文件
- `server/src/llm/structuredInvoke.ts` - 添加了特殊处理逻辑，当解析到数组时尝试包装成对象
